## Introduction

This write-up documents an original technique for executing arbitrary syscalls in pure C, without any call to `mmap`, `mprotect`, or any explicit allocation of executable memory. The technique relies on a fundamental and unavoidable property of compilation:

> **Any immediate value used in a C expression is encoded verbatim in the `.text` segment, which is by nature executable.**

By carefully choosing these immediate values, it is possible to make functional shellcode appear in that segment — not by writing bytes manually, but by constraining the compiler to produce them itself through the ordinary semantics of the language.

Three progressive versions are presented:

- **V1**: Execution via stack frame hijacking, using only local variable assignments.
- **V2**: Execution via dynamic scan of the `.text`, robust against all compiler optimizations.
- **V3**: Extension to complex syscalls (6 arguments, `r10` register) via a ROP gadget chain generated by immediates and orchestrated from the stack.

## Prerequisites

- x86-64 architecture, Linux
- System V AMD64 ABI — arguments passed in `rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9`
- Linux x86-64 syscall convention — arguments in `rax`, `rdi`, `rsi`, `rdx`, `r10`, `r8`, `r9`
- Basic knowledge of function prologue/epilogue in assembly
- Basic notion of Return-Oriented Programming (ROP)

## Context: ABI vs Syscall Divergence

A fundamental detail motivates the complexity of V3:

| Position | System V ABI | Linux Syscall |
|---|---|---|
| arg 4 | `rcx` | `r10` |

The `r10` register is never loaded by the ABI. For any syscall using a 4th argument (`mmap`, `openat`, etc.), the shellcode must explicitly load `r10` — something the libc does internally but the ABI does not.

## V1 — Shellcode via Pure Assignments

### Code

```c
void    function(int fd, char *str, unsigned long len)
{
    unsigned long    buffer[2];

    buffer[1] = 0x0007ebe589485552;
    buffer[1] = 0x06ebdc7d8b58016a;
    buffer[1] = 0x000008ebd0758b48;
    buffer[1] = 0xc3c9050fc8558b48;
    buffer[0] = buffer[5];
    buffer[5] = (unsigned long)function + 40;
    buffer[0] = buffer[0] + buffer[1];
}
```

### Injection via Immediate Value

Each assignment to `buffer[1]` forces the compiler to encode the constant into `.text`. The four values, decoded in little-endian, form a complete `write` shellcode:

```asm
; 0x0007ebe589485552
push rdx          ; save len
push rbp
mov  rbp, rsp
jmp  +7           ; skip null padding bytes

; 0x06ebdc7d8b58016a
push 0x1          ; SYS_write = 1
pop  rax
mov  edi, [rbp-0x24]   ; fd from caller's stack frame
jmp  +6

; 0x000008ebd0758b48
mov  rsi, [rbp-0x30]   ; str from stack frame
jmp  +8

; 0xc3c9050fc8558b48
mov  rdx, [rbp-0x38]   ; len from stack frame
syscall
leave
ret
```

The short `jmp` instructions skip the null bytes introduced by 64-bit constant encoding — the shellcode is continuous despite the padding.

### Return Address Hijacking

```c
buffer[0] = buffer[5];               // save the real return address
buffer[5] = (unsigned long)function + 40; // overwrite return address → shellcode
```

`buffer[5]` corresponds to the return address on the stack. On `ret`, execution jumps directly to the injected shellcode.

### Return Address Transport via `rdx`

```c
buffer[0] = buffer[0] + buffer[1];
```

To perform the addition, the compiler loads `buffer[1]` into `rdx`:

```asm
mov rax, [rbp-16]
mov rdx, [rbp-8]    ; rdx = saved real return address
add rax, rdx
mov [rbp-16], rax
```

`rdx` survives until `ret` without being overwritten. The shellcode reads it to resume normal execution after the syscall — with no explicit variable, no clean storage, only thanks to the persistence of an intermediate register value from an arithmetic operation.

### Limitations

- `buffer[5]` offset depends on the stack frame layout generated by the compiler.
- `function + 40` offset depends on the exact location of immediates in `.text`.
- Both values vary with compilation flags and compiler version.

## V2 — Dynamic Scan, Full Robustness

### Code

```c
long    write(int fd, char *buf, unsigned long len)
{
    unsigned long    ptr;
    long             result;

    ptr = (unsigned long)write;
    while (*(unsigned long *)ptr != 0xc3050f58016a)
        ptr++;
    result = ((long (*)(int, void *, unsigned long))ptr)(fd, buf, len);
    return (result);
}
```

### The Principle

The `while` condition compares a memory value against the constant `0xc3050f58016a`. To perform this comparison, the compiler generates:

```asm
mov rax, 0x0000c3050f58016a
cmp [ptr], rax
```

The constant is encoded in `.text` as (little-endian):

```
6a 01   →  push 0x1      ; SYS_write = 1
58      →  pop rax
0f 05   →  syscall
c3      →  ret
```

**The comparison constant is a complete `write` shellcode.** The compiler wrote it into an executable zone solely because the semantics of the comparison operation required it.

The scan locates this sequence from the start of the function, then calls it directly. The ABI places `fd` in `rdi`, `buf` in `rsi`, `len` in `rdx` — exactly what the syscall expects.

### Robustness

This version depends on no offset, no stack frame layout. The compiler must materialize the constant in `.text` regardless of optimization level — this behavior is **invariant** under `-O0`, `-O3`, `-fomit-frame-pointer`, or any other combination of flags.

## V3 — ROP Gadget Chain for Complex Syscalls

### Motivation

`mmap` requires 6 arguments, with the 4th going into `r10` — a register absent from the ABI. A single immediate sequence is no longer sufficient: a **gadget chain** is needed to progressively set up registers, using the stack as intermediate storage.

### General Architecture

```
C Code                    Stack                     Gadgets
─────────────────         ──────────────────        ──────────────────
setup1(args)         →    push addr, len,       →   gadget1
setup2(args)         →    push prot, flags,     →     └─ call → gadget2
gadget1(g2, g3, g4)  →    push fd, offset,          gadget2
                          push SYS_mmap                └─ call → gadget3
                                                    gadget3
                                                       └─ call → gadget4
                                                    gadget4
                                                    → unwind chain
                                                    → syscall in gadget1
```

### Full Code

```c
#include <stdio.h>

void	*mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset)
{
	size_t	result;
	size_t	gadget1;
	size_t	gadget2;
	size_t	gadget3;
	size_t	gadget4;
	size_t	setup;

	// Setup1: push SYS_mmap + addr + length + prot
	setup = (size_t)mmap;
	while (*(size_t *)setup != 0xC353525657096A5B)
		setup++;
	((void (*)(void *, size_t, int, int, int, off_t))setup)(addr, length, prot, flags, fd, offset);

	// Setup2: push flags + fd + offset
	setup = (size_t)mmap;
	while (*(size_t *)setup != 0xC35351415041515B)
		setup++;
	((void (*)(void *, size_t, int, int, int, off_t))setup)(addr, length, prot, flags, fd, offset);

	// Locate ROP gadgets
	gadget1 = (size_t)mmap;
	gadget2 = (size_t)mmap;
	gadget3 = (size_t)mmap;
	gadget4 = (size_t)mmap;
	while (*(size_t *)gadget1 != 0xC353050F58D7FF5B)
		gadget1++;
	while (*(size_t *)gadget2 != 0xC3505F5E5AD6FF58)
		gadget2++;
	while (*(size_t *)gadget3 != 0xC3575A41D2FF5F)
		gadget3++;
	while (*(size_t *)gadget4 != 0xC356584159415E)
		gadget4++;

	// Launch chain — gadget2/3/4 passed via ABI into rdi/rsi/rdx
	result = ((size_t (*)(size_t, size_t, size_t))gadget1)(gadget2, gadget3, gadget4);
	return ((void *)result);
}
```

### Gadget Decoding

**Setup1** — `0xC353525657096A5B`
```asm
pop  rbx        ; save continuation
push 0x9        ; SYS_mmap = 9
push rdi        ; addr
push rsi        ; length
push rdx        ; prot
push rbx        ; restore continuation
ret
```

**Setup2** — `0xC35351415041515B`
```asm
pop  rbx        ; save continuation
push rcx        ; flags
push r8         ; fd  (REX prefix — 2 bytes)
push r9         ; offset
push rbx        ; restore continuation
ret
```

**Gadget1** — `0xC353050F58D7FF5B` — entry point + syscall
```asm
pop  rbx        ; save continuation
call rdi        ; → gadget2  (rdi received via ABI)
pop  rax        ; SYS_mmap = 9 from stack
syscall
push rbx
ret
```

**Gadget2** — `0xC3505F5E5AD6FF58` — load rdi, rsi, rdx
```asm
pop  rax        ; save continuation
call rsi        ; → gadget3  (rsi received via ABI)
pop  rdx        ; prot
pop  rsi        ; length
pop  rdi        ; addr
push rax        ; restore continuation
ret
```

**Gadget3** — `0xC3575A41D2FF5F` — load r10 (flags)
```asm
pop  rdi        ; save continuation
call rdx        ; → gadget4  (rdx received via ABI)
pop  r10        ; flags  (REX prefix — 2 bytes)
push rdi        ; restore continuation
ret
```

**Gadget4** — `0xC356584159415E` — load r8, r9, rcx
```asm
pop  r14        ; save continuation  (REX — 2 bytes)
pop  rcx        ; fd
pop  r9         ; offset  (REX — 2 bytes)
pop  rsi        ; alignment padding
push r14        ; restore continuation
ret
```

### Execution Chain Diagram

```
gadget1(gadget2, gadget3, gadget4)
   │  rdi=gadget2, rsi=gadget3, rdx=gadget4
   │
   ├─ pop rbx          (save return to C caller)
   ├─ call rdi ────────────────────────────────────────┐
   │                                                   ▼
   │                                              gadget2
   │                                          pop rax  (save return to gadget1)
   │                                          call rsi ──────────────────────────────┐
   │                                                                                 ▼
   │                                                                            gadget3
   │                                                                        pop rdi  (save return to gadget2)
   │                                                                        call rdx ──────────────────┐
   │                                                                                                   ▼
   │                                                                                              gadget4
   │                                                                                          pop r14
   │                                                                                          pop rcx   ← fd
   │                                                                                          pop r9    ← offset
   │                                                                                          push r14
   │                                                                                          ret ──────┘
   │                                                                        pop r10   ← flags
   │                                                                        push rdi
   │                                                                        ret ───────────────────────┘
   │                                          pop rdx   ← prot
   │                                          pop rsi   ← length
   │                                          pop rdi   ← addr
   │                                          push rax
   │                                          ret ───────────────────────────────────┘
   ├─ pop rax          ← SYS_mmap = 9
   ├─ syscall          rax=9, rdi=addr, rsi=length, rdx=prot, r10=flags, r8=fd, r9=offset
   ├─ push rbx
   └─ ret              → return to C caller
```

### Chain Properties

Each gadget follows the structure:

```asm
pop  reg_save    ; 1 byte  — save continuation before call pollutes stack
call reg_next    ; 2 bytes — jump to next gadget
[action]         ; 3 bytes max — pops into syscall registers
push reg_save    ; 1 byte  — restore continuation
ret              ; 1 byte  — resume return flow
```

**Budget per gadget: 8 bytes.** Extended registers (`r8`-`r15`) cost 2 bytes (REX prefix), constraining gadgets that need them to reduce their action to 1 remaining byte.

The stack acts as an **extended register file**: argument values are deposited by the setups, consumed in order by the gadgets via `pop`, with no register shared between gadgets for data transport.

## Relationship to Existing Techniques

**Return-Oriented Programming (ROP)** — classic ROP *searches* for gadgets in an existing binary. Here, gadgets are *deliberately generated* by the compiler via constants chosen for their binary encoding.

**Data-Only Attacks** — modifying data without injecting code. Here, the code already exists in `.text`, passively materialized by the compiler.

**Compiler-assisted exploitation** — exploiting the deterministic behavior of the compiler. The attack surface is not a memory vulnerability but the mandatory semantics of the C language.

The key distinction from all these techniques: **no code byte is manually written to memory, no memory protection is bypassed.** The shellcode and gadgets emerge as unavoidable side effects of compiling syntactically legitimate C expressions.

---

## Conclusion

These three techniques form a coherent progression around a single principle: **the compiler, constrained by language semantics, produces exactly the desired bytes in executable zones, without any awareness of doing so.**

V3 demonstrates that this approach scales to arbitrarily complex syscalls. By combining immediate generation, dynamic scanning, and a gadget chain orchestrated from the stack, it is possible to execute any Linux x86-64 syscall from pure C — with no allocated executable memory, no manually written bytes, no bypassed protections.

The attack vector is the compiler itself. The technique is called **Semantic Shellcode Injection**.
